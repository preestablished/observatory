# 07 — M2 (part 1): `obs-scrape` Prometheus Parser + `metrics_raw`

## Goal

New crate `obs-scrape`: text-format Prometheus parser (counters, gauges,
histograms), static target list from config, staleness tracking, samples
batched into `metric_series`/`metrics_raw` through the single writer.
Owner accept-when list: IMPLEMENTATION-PLAN §M2 (parser/staleness bullets).
Depends on package 03 only — can run parallel to 05/06. No live sister
services exist yet; correctness is proven on checked-in fixtures (that is
the documented M2 posture: "until real services exist, fixtures come from
their docs' metric lists + the `prometheus` crate's own output").

## Design

- **Parser** (pure module, no I/O — unit-testable): text exposition format;
  `# TYPE` lines drive handling — `counter`/`gauge` → one sample per line;
  `histogram` → `_bucket{le=…}`, `_sum`, `_count` samples kept as separate
  series (labels canonicalized; grain math and p50/p99 derivation happen in
  package 08); `summary` and untyped lines → store as gauges (don't drop
  data silently); comments/blank lines skipped; malformed lines counted +
  skipped, never a whole-scrape failure. Label set → canonical sorted-key
  JSON (ARCHITECTURE §3.1 `metric_series.labels`).
- **Series interning**: `metric_series` lookup/insert cache keyed
  `(service, metric, labels)`; `service` = target `name` from config
  (ARCHITECTURE: "names must match the service column used in series
  keys").
- **Scrape loop**: tokio task per config (`[scrape] interval`, default 5s;
  `[[scrape.targets]] {name, url}`); HTTP GET with timeout ~= interval/2;
  each scrape → one `WriteBatch::MetricSamples` (ts from `Clock`, one ts
  per scrape pass). Use a plain `reqwest`-or-`hyper` GET (reqwest with
  rustls is already sanctioned by ARCHITECTURE §1 for webhooks).
- **Staleness tracking**: per-target `last_success_ns` +
  `obs_scrape_failures_total{target}`; expose per-target staleness (an
  in-memory map + gauge `obs_scrape_age_seconds{target}`) — M4's `absent`
  alert rule consumes this; only the tracking lands now.
- Retention note: `metrics_raw` is 24 h retention **in M8** — no sweeper
  now; document in code.

## Fixtures (checked in under `crates/obs-scrape/tests/fixtures/`)

One golden body per sister service, hand-written from the metric-name
contract in observatory ARCHITECTURE §3.3 / INTEGRATION §2:
`determinism-hypervisor` (`dh_slots_total`, `dh_slots_busy`,
`dh_jobs_completed_total`, `dh_fork_restore_seconds` histogram,
`dh_determinism_failures_total`), `snapshot-store` (`ss_*` incl.
`ss_disk_used_bytes`/`ss_disk_total_bytes`), `exploration-orchestrator`
(`eo_expansions_total`, `eo_commits_total`, `eo_duplicates_total`,
`eo_frontier_size`, `eo_best_score`, `eo_inflight_jobs`, `eo_stall_level`),
`state-scorer` (`sc_*` incl. `sc_gpu_batch_latency_seconds` histogram),
`input-synthesizer` (`is_*`), `replay-renderer` (`rr_*`), `control-plane`
(`cp_*`) — plus one fixture generated from the `prometheus` crate's own
encoder (a real library-emitted body, capturing escaping/format quirks).
Expected-parse results are hand-written golden Rust/JSON tables next to
each fixture (never generated by the parser under test).

## Acceptance (exact commands)

```bash
cargo test -p obs-scrape                 # golden + unit
cargo test --workspace
# integration: static stub server (tiny axum serving a fixture body) scraped
# end-to-end into a tempfile DB:
cargo test -p obs-scrape --test scrape_end_to_end
```

- Every fixture parses to its golden table exactly (names, labels order
  canonicalized, values, histogram series split).
- Malformed-line fixture: good lines land, bad lines counted, scrape
  succeeds.
- Target-down test: failures metered, `last_success_ns` frozen, loop keeps
  ticking, `/healthz` unaffected.
- Repeated scrape of identical body → new rows at new timestamps (gauges
  are time series, not upserts) — assert row counts over two ticks.

## Failure guidance

- Do not reach for a full Prometheus-client parsing dependency; the text
  format subset above is ~200 lines and the repo's dependency posture is
  deliberate (ARCHITECTURE §1 list). If escaping edge cases bite (label
  values with `\n`/`\"`), extend the fixture set first, then the parser.
- If `metrics_raw` PK collisions appear (`(series_id, ts_ns)` with two
  targets sharing a series): the interning key is wrong — service name must
  be part of the series identity; never widen the PK.
- Histogram fixtures are the classic trap: `le` label is a **string**
  (`"0.5"`, `"+Inf"`); keep it verbatim in the canonical label JSON, don't
  parse to float for identity.
